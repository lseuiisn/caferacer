# 드라이브 카페 추천 플랫폼 MVP 설계

## 1. 확정된 요구사항

| 항목 | 결정 |
| --- | --- |
| 서비스 지역 | 서울, 경기도 |
| 초기 카페 수 | 크롤링 후 운영자가 검수한 20개 |
| 계정 | 필수, 카카오 및 구글 로그인 |
| 위치 기록 | 필수 동의 후 드라이브 기록에 사용 |
| 경로 | TMAP 자동차 경로 API로 시간·거리 산정, 최종 길안내는 TMAP 앱으로 실행 |
| 앱 | Flutter + Riverpod |
| 서버 | FastAPI + SQLAlchemy + MySQL |

## 2. MVP 사용자 흐름

1. 사용자는 카카오 또는 구글로 로그인하고 위치 권한 및 위치정보 수집 동의를 선택한다.
2. 사용자는 현재 위치, 희망 왕복 시간, 분위기와 카페 조건을 입력한다.
3. 서버는 후보 코스·카페를 거리로 1차 필터링한 뒤 TMAP 경로 시간으로 재검증하고, 상위 3개를 반환한다.
4. 사용자는 코스를 선택해 상세 정보를 보고 TMAP으로 길안내를 실행한다.
5. 출발 후 앱은 사용자가 기록 시작을 명시적으로 선택한 경우에만 GPS 포인트를 저장한다.
6. 도착 또는 기록 종료 후 이동거리·시간·평균속도를 계산해 저장한다.

## 3. 추천 전략

초기에는 자동 경로 생성보다 검수된 코스와 카페의 조합을 추천한다. 추천 서비스는 다음 순서로 후보를 좁힌다.

1. 서울·경기도의 활성 코스와 카페만 선택한다.
2. 사용자의 위치 및 왕복 시간으로 직선거리 기반 1차 후보를 제한한다.
3. 상위 후보에만 TMAP 경로 API를 호출해 출발지-경유지-카페-출발지 예상 시간을 확인한다.
4. 시간 예산을 만족하는 후보를 아래 점수로 정렬한다.

`총점 = 시간 적합도 35 + 드라이브 적합도 25 + 조건 일치도 20 + 카페 품질 15 + 운영 보정 5`

각 하위 점수와 사용한 경로 시간은 추천 결과에 저장한다. 이후 실제 완주 기록을 이용해 가중치를 조정할 수 있다.

## 4. 외부 데이터 및 운영 원칙

- TMAP API 키는 Flutter 앱에 넣지 않고 FastAPI 서버 환경 변수로만 관리한다.
- TMAP 경로·지오코딩 API 응답은 제공자 약관이 허용하는 보존 기간 안에서만 짧게 캐싱한다. 같은 추천 요청의 중복 호출을 막되, TMAP에서 받은 좌표·경로 데이터를 영구 카페 데이터로 저장하지 않는다.
- 카페 기본 정보는 공공 데이터와 공식 카페 채널을 우선 사용한다. 크롤링 데이터는 운영자 검수 전에는 공개하지 않는다.
- 외부 플랫폼의 사진·리뷰·평점을 복제해 저장하거나 노출하지 않는다. 서비스 리뷰는 자체 작성 리뷰만 저장한다.
- 각 수집 값의 출처 URL, 수집 시각, 최종 검수 시각을 남긴다.
- 위치 원본은 사용자의 명시적 기록 시작 후에만 저장하고, 탈퇴 시 삭제·익명화 정책을 적용한다.

```text
project/

│       ├── features/
│       │   ├── auth/
│       │   ├── cafes/
│       │   ├── recommendations/

│       │   ├── favorites/
│       │   └── rankings/

├── backend/
│   └── app/
│       ├── api/routers/
│       ├── core/             # 설정, 보안, DB 세션
│       ├── models/
│       ├── repositories/
│       ├── schemas/

```

## 6. 구현 순서

1. MySQL 스키마와 Alembic 마이그레이션
2. 소셜 로그인 및 JWT 인증
3. 카페·코스 조회 API와 20개 시드 데이터
4. 추천 API 및 TMAP 경로 어댑터
5. Flutter 로그인·검색·추천 결과 화면
6. TMAP 실행과 GPS 기록
7. 즐겨찾기·리뷰·랭킹

## 7. 향후 제안

- 운영자용 카페·코스 검수 화면을 별도 기능으로 둔다. 초기에는 DB 관리 도구로 대체 가능하다.
- 추천 실패 시 빈 화면 대신 시간 범위를 늘린 대체 코스와 조건 완화 이유를 제공한다.
- 백그라운드 GPS는 배터리와 개인정보 부담이 크므로, 기본값은 경로 요약 기록으로 하고 원본 포인트 기록은 사용자 선택으로 제공하는 방안을 제안한다.
